# VI Cloud Control Plane

VI supports two operating modes. Both use the same dashboard UI and the same vi-agent binary — only the environment variables differ.

## Live deployment

| Item | Value |
|------|-------|
| Relay URL | `https://vi-relay-jozzy.fly.dev` |
| Fly app | `vi-relay-jozzy` |
| Region | `sin` |
| VM | shared-cpu-1x, 256 MB |
| Volume | `vi_relay_data`, 1 GB, encrypted |
| Last E2E verified | 2026-05-18 |

**E2E verified:** health ✓, auth (old token 401, new token 200) ✓, dashboard cloud mode ✓, enrollment pair command ✓, vi-agent connect ✓, Claude session ✓, browser terminal echo ✓.

---

## Mode 1: Local / Same-Wi-Fi (default)

No relay server needed. Dashboard and vi-agents must be on the same network.

```
vi-agent  ──→  dashboard (port 3000)
browser   ──→  dashboard (port 3000)
terminal  ──→  dashboard (port 14801)
```

**Setup:**
```bash
# packages/web/.env.local  (no relay vars needed)
VI_ACCESS_TOKEN=your-dashboard-password
```

**Pairing command (generated by dashboard):**
```
vi-agent pair --server http://192.168.x.x:3000 --code XXXX --start
```

---

## Mode 2: Cloud Control Plane

VI Relay is a persistent cloud server that both the dashboard client and vi-agents connect to outbound. Neither side needs a fixed IP or open inbound port.

```
vi-agent  ──→  VI Relay (public)  ←──  dashboard client
browser   ──→  dashboard (local)   ──→  VI Relay API
terminal  ──→  VI Relay /vi-agent-relay  ←──  dashboard
```

### First-time deploy (Fly.io)

> **Volume is required.** Without it the SQLite DB is wiped on every deploy. Create the volume first, then deploy — never in the other order.

```bash
cd packages/relay

# 1. Create persistent volume BEFORE first deploy
fly volumes create vi_relay_data --size 1 --region sin --app vi-relay-jozzy

# 2. Deploy (no HA, cost-minimised)
fly deploy --ha=false --app vi-relay-jozzy

# 3. Set secrets (generate real tokens — do NOT use these placeholders)
#    Generate: openssl rand -hex 32  (run twice)
fly secrets set \
  VI_RELAY_TOKENS="<DAEMON_TOKEN>:daemon:local-daemon,<VI_TOKEN>:pi:local-pi" \
  VI_RELAY_OWNER_TOKEN="<VI_TOKEN>" \
  VI_RELAY_PUBLIC_WS_URL="wss://vi-relay-jozzy.fly.dev" \
  --app vi-relay-jozzy
```

**Verify deploy:**
```bash
curl https://vi-relay-jozzy.fly.dev/health        # → {"status":"ok"}
curl -s -o /dev/null -w "%{http_code}" https://vi-relay-jozzy.fly.dev/presence  # → 401
```

### Configure the dashboard (cloud mode)

```bash
# packages/web/.env.local  — DO NOT COMMIT THIS FILE
VI_RELAY_BASE_URL=https://vi-relay-jozzy.fly.dev
VI_RELAY_VI_TOKEN=<redacted>
VI_RELAY_DAEMON_TOKEN=<redacted>
VI_RELAY_PUBLIC_WS_URL=wss://vi-relay-jozzy.fly.dev
```

`.env.local` is gitignored. Never stage or force-add it. Never paste real token values in chat, logs, or code review.

When `VI_RELAY_BASE_URL` + `VI_RELAY_VI_TOKEN` are both set, the dashboard automatically uses the relay as its backend. Local mode is the fallback when either is missing.

### Pairing a new machine (cloud mode)

The dashboard generates relay-aware pair commands automatically. In cloud mode the pair command uses the relay URL:

```
vi-agent pair --server https://vi-relay-jozzy.fly.dev --code XXXX --start
```

vi-agent stores the relay URL and token in its state file after pairing. All subsequent heartbeats and reports go to the relay directly — the dashboard is not in the loop for those.

## Token rotation

Rotate tokens when: a token is accidentally exposed (printed in terminal, pasted in chat, appears in a git diff or log).

**Steps:**

1. Generate new tokens locally — do not paste them here:
   ```bash
   openssl rand -hex 32   # → NEW_DAEMON_TOKEN
   openssl rand -hex 32   # → NEW_VI_TOKEN
   ```

2. Set new Fly secrets (you run this with real values):
   ```bash
   fly secrets set \
     VI_RELAY_TOKENS="<NEW_DAEMON_TOKEN>:daemon:local-daemon,<NEW_VI_TOKEN>:pi:local-pi" \
     VI_RELAY_OWNER_TOKEN="<NEW_VI_TOKEN>" \
     VI_RELAY_PUBLIC_WS_URL="wss://vi-relay-jozzy.fly.dev" \
     --app vi-relay-jozzy
   ```
   Fly triggers an automatic redeploy. Wait for it to complete.

3. Update `packages/web/.env.local`:
   ```
   VI_RELAY_VI_TOKEN=<NEW_VI_TOKEN>
   VI_RELAY_DAEMON_TOKEN=<NEW_DAEMON_TOKEN>
   ```

4. Restart the dashboard dev server.

5. Re-pair all vi-agents — the daemon token in each agent's state file is now invalid:
   ```bash
   # In dashboard: create a new enrollment, copy the pair command
   vi-agent pair --server https://vi-relay-jozzy.fly.dev --code XXXX --start
   ```

6. Verify (status codes only — no token values):
   - Old token → 401
   - New token → 200
   - Overview reads from relay (`generatedAt` field present)
   - New enrollment pair command uses Fly URL

---

## Token format

`VI_RELAY_TOKENS` is a comma-separated list of `token:kind:label` entries:
```
daemon-abc:daemon:my-machines,pi-xyz:pi:dashboard
```

- `daemon` tokens: used by vi-agent (heartbeat, register, report, enrollment consume)
- `pi` tokens: used by the dashboard (overview, jobs, approvals, enrollments)

## Architecture summary

| Route | Caller | Mode |
|-------|--------|------|
| `POST /v1/daemon/register` | vi-agent | cloud relay |
| `POST /v1/daemon/heartbeat` | vi-agent | cloud relay |
| `GET /v1/daemon/agents/:id/poll` | vi-agent | cloud relay |
| `POST /v1/daemon/jobs/report` | vi-agent | cloud relay |
| `POST /v1/daemon/requests` | vi-agent | cloud relay |
| `POST /v1/daemon/enrollments/consume` | vi-agent pair | cloud relay |
| `GET /v1/vi/overview` | dashboard | cloud relay |
| `POST /v1/vi/jobs` | dashboard | cloud relay |
| `POST /v1/vi/approvals/:id/respond` | dashboard | cloud relay |
| `GET /api/remote-agents/*` | vi-agent (local) | local dashboard |
| `ws://../vi-agent-relay` | vi-agent terminal | relay WebSocket |

**`VI_RELAY_VI_BASE_URL` is deprecated.** The relay no longer calls back into the local dashboard. Remove it from any existing configs.

## SQLite notes

The relay uses SQLite (WAL mode) as its persistent store. At 100 agents heartbeating every 5s, write throughput is ~20 writes/sec — well within SQLite's limits.

To migrate to Postgres later, only the DB layer (`packages/relay/src/db.ts` + `store.ts`) needs to change. The API surface stays the same.
